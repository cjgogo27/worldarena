# 1. 项目总览

## 1.1 当前目标
围绕人类移动性建模，设计并实现一个以开源大模型为 backbone 的统一 mobility foundation model，并通过 skill 机制增强其推理、补全、生成和预测能力。

## 1.2 当前最优方向
不建议只做“多任务 mobility 基座模型”，因为这条线已经在 2024–2025 明显变热。  
更有竞争力的方向是：

- 使用 **Open LLM backbone（Qwen3-8B）**
- 引入 **mobility-specific structured encoder**
- 将外部知识、检索、规则、上下文整合为 **skills**
- 统一四个任务：next POI / prediction / completion / generation
- 研究 **skill augmentation + optional skill distillation**

## 1.3 当前推荐题目风格
可选题目方向：
- **Skill-Augmented Mobility Foundation Model with an Open LLM Backbone**
- **From Tools to Skills: Building an Open Mobility Foundation Model for Human Mobility**
- **A Unified Skill-Augmented Mobility LLM for Next POI, Prediction, Completion, and Generation**
- **SkillMove-LLM: Structured Skill Augmentation for Open Mobility Foundation Models**

# 2. 为什么原始想法不够新，以及如何升级

## 2.1 原始思路
原始想法是：
- 路线一：人类移动性 agent + 工具
- 路线二：人类移动性大模型基座（多任务）

这两个想法都成立，但都还不够“论文化”。

## 2.2 为什么“只做多任务基座”不够
原因：
1. 2024 已经有 PMT 这类工作做 unified mobility foundation model
2. 2025 又有 UniTraj、MoveGPT、cross-domain mobility foundation model 等继续推进
3. 如果只说“我是多任务统一基座”，会被认为创新不足

## 2.3 为什么“只做 agent + tools”也不稳
原因：
1. agent 的效果容易过度依赖 LLM 与 prompt
2. tools 的可复现性、稳定性、成本和依赖会被审稿人质疑
3. 如果工具只是简单拼接，方法贡献不强

## 2.4 升级方向
最终升级为：

**Foundation Backbone + Skill Abstraction + Multi-task Mobility Modeling**

关键变化：
- 不把“tool 调用”当方法核心
- 将 tools 封装为 skills
- backbone 不做普通 transformer，而用 open LLM backbone
- 多任务不再是唯一卖点，而是统一框架的一部分

# 3. 最终建议的方法框架

## 3.1 总体框架
建议最终方法命名为：

**Skill-Augmented Mobility Foundation Model**

框架由 4 层组成：

1. **Mobility Input Layer**
2. **Structured Mobility Encoder / Projector**
3. **Open LLM Backbone（Qwen3-8B）**
4. **Skill Layer + Multi-task Heads**

## 3.2 输入
输入是轨迹序列，每个 event 至少包含：

- user_id
- timestamp
- lat / lon
- location_id 或 grid_id
- category_id
- city_id
- dataset_id

## 3.3 结构化事件表示
每个 step 不是普通文本，而是结构化 mobility event：

- loc_token
- grid_token
- category_token
- hour_token
- dow_token
- gap_token
- city_token
- dataset_token

建议先 embedding 化，再投影到 Qwen 的 token space。

## 3.4 backbone
主 backbone 推荐：
- **Qwen3-8B**

原因：
- 你已有本地模型
- 开源、可训练、可微调、可复现
- 比闭源产品更适合作为论文主 backbone

## 3.5 多任务 heads
建议挂四类 heads：

1. **Next POI Head**
2. **Trajectory Prediction Head**
3. **Trajectory Completion Head**
4. **Trajectory Generation Head**

## 3.6 skill 层
skill 层不直接写成“若干工具”，而是写成面向 mobility 的可复用过程模块。

建议至少包括：

1. **Personal Routine Skill**
2. **Collective Mobility Skill**
3. **Urban Context Skill**
4. **Feasibility Reasoning Skill**
5. **Gap-Recovery Skill**

## 3.7 融合方式
建议优先从简单到复杂：

### 方案 A：Skill Summary Tokens
将 skill 输出编码成结构化 summary token，作为 prefix 输入 backbone。

### 方案 B：Skill Memory Bank
将 skill 输出编码成 memory bank，通过 cross-attention 与 backbone hidden states 融合。

建议第一版先做 A。

## 3.8 可选增强：Skill Distillation
训练时调用 skill；推理时部分 skill 能力蒸馏回 backbone。  
这部分是一个很值得打的创新点。

# 4. backbone 选择：Qwen / 开源模型 / 闭源模型的取舍

## 4.1 backbone 需要训练吗
需要，但不一定要从零预训练。

建议训练方式：
- structured mobility encoder / projector
- Qwen 的 LoRA 微调
- 或部分层微调
- 再做多任务 supervised tuning
- 再做 skill augmentation tuning

## 4.2 能否直接用 ChatGPT 这类闭源模型做主 backbone
不建议作为论文主 backbone。

原因：
1. 难以复现
2. 权重不可得
3. 模型版本会变
4. 消融难做
5. 训练与部署不可控
6. 评审更难信服

更合适的用法：
- teacher
- planner baseline
- evaluator / judge
- distillation assistant

## 4.3 为什么 Qwen3-8B 更适合
你当前已有路径：

`/data/alice/cjtest/FinalTraj_arr/finetune/models/Qwen3-8B`

它的优势：
- 可控
- 开源
- 支持本地训练
- 容量适中
- 可作为论文方法主体

## 4.4 是否需要两个模型
可以，但要主次分明。

推荐方案：
- **主模型**：Qwen3-8B student backbone
- **辅模型（可选）**：更强闭源或更大开源模型作为 teacher / planner / distillation helper

注意：
论文主方法必须落在可复现的主模型上。

# 5. skill 与 tool 的关系

## 5.1 为什么要从 tool 升级到 skill
因为 tool 只是原子能力，skill 才是方法级单位。

### Tool
例如：
- 路径距离计算
- POI 检索
- 天气查询
- 相似轨迹召回

### Skill
例如：
- Personal Routine Skill
- Collective Mobility Skill
- Urban Context Skill
- Feasibility Reasoning Skill
- Gap-Recovery Skill

skill = 工具 + 规则 + 输入输出 schema + 触发条件 + 融合策略

## 5.2 在论文里的表述建议
不要写成：
- “我们接入了很多免费 API”
- “我们用了很多 external tools”

建议写成：
- “We introduce a skill abstraction for modular mobility reasoning.”
- “Skills provide reusable structured external knowledge and reasoning processes.”
- “Each skill encapsulates tools, priors, constraints, and retrieval logic.”

## 5.3 是否还要强调免费 API
不需要作为论文主卖点。  
可以在实现部分弱化表述为：
- public resources
- accessible context modules
- reproducible external context interfaces

# 6. 多任务任务定义与边界

## 6.1 任务集合
本文建议统一做四个任务：

1. **Next POI**
2. **Trajectory Prediction**
3. **Trajectory Completion**
4. **Trajectory Generation**

## 6.2 必须避免 prediction 和 generation 概念混淆
建议定义：

### Next POI
给历史轨迹，预测下一步位置/POI。

### Trajectory Prediction
给历史轨迹，预测未来 k 步真实轨迹，强调监督 forecasting。

### Trajectory Completion
给定局部缺失轨迹，补全中间或局部被 mask 的片段。

### Trajectory Generation
给定部分条件，生成较长未来轨迹，强调分布真实性、多样性和模式建模，而不只是点对点命中。

## 6.3 输入输出
### 输入
- 轨迹历史
- 时间信息
- 区域或类别信息
- 可选 skill context

### 输出
- Next POI：下一点预测
- Prediction：未来 k 步序列
- Completion：缺失片段
- Generation：长轨迹

# 7. 数据方案设计

## 7.1 当前数据方向
你提到的核心数据是：

- 四个 Foursquare++
- ISP 数据

## 7.2 数据统一目标
统一为同一 schema：

- user_id
- timestamp
- lat
- lon
- location_id
- category_id
- city_id
- dataset_id

## 7.3 数据预处理步骤
1. 去重
2. 时间排序
3. 去异常点
4. 统一时间格式
5. 过滤过短轨迹
6. 空间离散化
7. 构建 train / val / test split

## 7.4 空间表示建议
建议至少做两级表示：

### 方案一：网格主表示
- H3 / Geohash / 自定义网格

### 方案二：双层表示
- coarse grid
- fine POI / region token

建议先从方案一开始，降低复杂度。

## 7.5 时间表示建议
每个事件建议包括：
- hour of day
- day of week
- weekend / weekday
- time gap bucket

## 7.6 划分建议
必须时间切分，避免泄漏：

- train：前 70%
- val：中间 10%
- test：后 20%

或按用户内时间分割。

## 7.7 潜在数据问题
1. 数据粒度不一致
2. ISP 与 Foursquare++ 分布差异大
3. 类别体系不统一
4. 长尾地点严重
5. 不同城市差异大
6. 可能出现未来信息泄漏
7. 轨迹序列长度不均衡

## 7.8 输入输出与决策点
### 输入
原始轨迹数据

### 输出
统一格式数据、切分数据、训练样本序列

### 需要用户决策
1. 是否统一为网格 token 主表示
2. 是否保留 POI category
3. 是否做跨城市统一训练
4. 是否加入 ISP 作为同一 backbone 的训练数据

### 备选方案
如果 ISP 与 Foursquare++ 差异过大，可以先：
- 先用 Foursquare++ 做主实验
- ISP 做扩展实验或 cross-domain test

# 8. baseline 与相关工作整理

## 8.1 总体原则
对比实验不能只做一种类型。  
最稳的结构是三层：

1. **任务内 baseline**
2. **foundation / unified baseline**
3. **agent / skill baseline**

## 8.2 新近且最重要的基线
### A. Foundation / Unified
1. **PMT (2024)**
   - 价值：最直接的 mobility foundation baseline
   - 任务：next-location / imputation / generation
   - 优先级：必比

2. **UniTraj (2024/2025)**
   - 价值：universal trajectory foundation model
   - 优先级：强烈建议比

3. **MoveGPT (2025)**
   - 价值：新的 mobility FM + scaling / MoE 方向
   - 优先级：建议比，至少在 related work 重点讨论

### B. Agent / LLM / Skill
1. **AgentMove (2024, NAACL 2025 Main)**
   - 价值：agent 路线最自然的对比对象
   - 优先级：必比

2. 其他 2025–2026 mobility agent
   - 后续继续补充
   - 若开源可复现，再列入实验

## 8.3 经典任务 baseline
### Next POI / next-location
- DeepMove
- GETNext
- STAN
- LSTPM

保留 1–2 个即可。

### Completion / Imputation
- KAMEL
- 更近的 trajectory imputation baseline（后续继续补）

### Generation
- TrajGDM
- 其他 diffusion / generation baseline（后续继续补）

## 8.4 baseline 复现策略
必须先做表格管理：

字段建议：
- baseline 名称
- 年份
- 论文链接
- GitHub 链接
- 任务
- 是否开源
- 是否能直接复现
- 输入格式
- 改造难度
- 预计优先级

## 8.5 baseline 复现优先级
### 第一阶段
- PMT
- AgentMove
- 1 个 next POI baseline

### 第二阶段
- UniTraj
- 1 个 completion baseline
- 1 个 generation baseline

### 第三阶段
- MoveGPT 或其他更新方法
- 更多 agent / skill baseline
- 更强多任务方法

# 9. 对比实验设计

## 9.1 不能只做一种对比
不能只做：
- 每个任务对各自 baseline
也不能只做：
- foundation model 对比

必须两者都做。

## 9.2 推荐三层对比
### 层 1：任务内 baseline
每个任务都与对应强 baseline 比。

### 层 2：foundation baseline
与 PMT / UniTraj / MoveGPT 等比。

### 层 3：agent / skill baseline
与 AgentMove、你自己的 no-skill 版本、backbone-only 版本比。

## 9.3 消融实验建议
至少包括：
1. w/o skills
2. w/o Personal Routine Skill
3. w/o Collective Mobility Skill
4. w/o Urban Context Skill
5. w/o Feasibility Skill
6. w/o skill selector
7. static skill fusion vs learned fusion
8. no distillation
9. single-task vs multi-task
10. Qwen backbone vs smaller backbone

## 9.4 评测指标
### Next POI
- Acc@1
- Acc@5
- MRR
- NDCG

### Trajectory Prediction
- step accuracy
- ADE / FDE（若用坐标）
- sequence consistency

### Trajectory Completion
- masked token accuracy
- spatial error
- top-k hit

### Trajectory Generation
- distribution matching
- jump length KL
- radius of gyration error
- visitation frequency correlation
- diversity
- exploration / return ratio

## 9.5 什么结果算成功
### 阶段 1 成功标准
- backbone 能稳定收敛
- 多任务结果不显著差于单任务基线
- 至少在 1–2 个任务优于 backbone-only baseline

### 阶段 2 成功标准
- 加入 skill 后比 no-skill 稳定提升
- 至少一个任务对强 baseline 有明显优势

### 阶段 3 成功标准
- 多任务整体优于 foundation baseline
- skill / distillation 带来解释性与性能收益

## 9.6 结果不好时如何调整
1. 降低任务数，先保 next POI + completion
2. 先用 Qwen backbone + summary token skill 融合
3. 暂缓 distillation
4. ISP 先不并入主训练，只做外部测试
5. 先以单城 / 单数据集验证框架有效性
